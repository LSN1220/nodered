# Node-RED 后端开发学习路线

## 1. 文档定位

这份文档是你的总览目录，目标不是一次性讲完所有知识点，而是明确：

- 学什么
- 先后顺序是什么
- 每个阶段要做什么项目
- 需要补哪些基础
- 后续样例代码应该如何组织

你的学习目标已经收敛为以下方向：

- 以项目实战为主
- 主线项目以 `管理后台 API` 和 `物联网 / 设备数据平台` 为核心
- `JavaScript` / `Node.js` / `Express` / 数据库设计需要重点深入
- 环境部署依赖 `Docker` 和 `docker-compose`
- 后续每个知识点都要配套可直接运行的独立样例项目

---

## 2. 学习总策略

推荐采用 `双主线项目驱动 + 基础穿插强化` 的方式。

原因很直接：

- 你目前是 `JavaScript / Node.js` 初学者，如果只学理论，反馈慢，容易失去上下文
- 如果只学 `Node-RED`，虽然上手快，但后续做真正的后端服务会缺少代码分层、接口设计、数据库建模能力
- 双主线可以让你同时掌握 `流程编排能力` 和 `代码型后端能力`

因此整个学习路线分为两条主线：

- `主线 A：管理后台 API`
  - 目标：学会构建标准后端服务
  - 重点：`Express`、REST API、认证授权、CRUD、日志、上传、分页、过滤、数据库设计
- `主线 B：物联网 / 设备数据平台`
  - 目标：学会使用 `Node-RED` 处理真实业务流
  - 重点：HTTP 接入、MQTT、设备数据处理、规则引擎、告警、数据落库、可观测性

数据库设计作为独立强化项，不附属于某一条主线，而是贯穿整个路线。

---

## 3. 你最终要具备的能力

完成这条路线后，你应该能独立完成以下事情：

- 使用 `Docker` / `docker-compose` 搭建本地后端开发环境
- 理解 `Node.js` 的运行方式、模块系统、异步模型、包管理
- 用 `Express` 编写结构清晰的后端 API
- 设计基础可用的关系型数据库表结构
- 理解索引、约束、事务、查询优化的基本原则
- 使用 `Node-RED` 编排 HTTP、数据库、MQTT、定时任务和业务规则
- 把 `Node-RED` 作为流程层，把 `Express` 作为代码层协同使用
- 独立搭建一个管理后台后端
- 独立搭建一个简单的物联网数据接入与处理平台

---

## 4. 学习阶段总览

建议按 `14 周` 推进。这个节奏适合初学者，也能保证每一阶段都有产出。

### 阶段 0：环境与认知准备（第 1 周）

目标：

- 理解整个学习路线涉及哪些技术
- 跑通本地 `Docker` 开发环境
- 知道 `Node-RED`、`Node.js`、`Express`、数据库各自负责什么

要完成的事情：

- 安装并验证 `Docker`
- 安装并验证 `docker-compose`
- 跑通当前仓库中的 `Node-RED` 环境
- 进入 `Node-RED` 管理界面，熟悉节点、流、部署方式
- 建立最基础的后端技术地图

你此时不追求“会写”，只追求“知道每个组件在系统里做什么”。

### 阶段 1：JavaScript / Node.js 入门补课（第 2-3 周）

目标：

- 建立写后端代码所需的最小基础

重点知识：

- 变量、函数、对象、数组
- 模块化：`CommonJS` 和 `ESM`
- 异步：回调、`Promise`、`async/await`
- `npm`、`package.json`
- Node.js 内置模块
- 文件读写
- 简单的 HTTP 服务概念

产出：

- 若干个可运行的独立样例项目
- 一个最小 Node.js HTTP 示例

### 阶段 2：Express 后端基础（第 4-5 周）

目标：

- 能写出基础管理后台 API

重点知识：

- 路由
- 中间件
- 请求参数校验
- 错误处理
- 控制器与服务层拆分
- REST API 设计基础
- `Postman` / `curl` 调试

产出：

- 一个可运行的 `Express` 后端样例
- 一个基础用户管理 API

### 阶段 3：数据库设计基础（第 6-7 周）

目标：

- 把“会建表”提升到“能为业务建合理的表”

重点知识：

- 实体、属性、关系
- 一对一、一对多、多对多
- 主键、外键、唯一约束
- 索引的作用与基本策略
- `CRUD` SQL
- 分页、排序、过滤
- 事务和一致性基础
- 软删除、审计字段、状态字段

产出：

- 管理后台项目数据库设计初稿
- 物联网平台数据库设计初稿
- 若干数据库建模样例项目

### 阶段 4：管理后台 API 主线项目（第 8-10 周）

目标：

- 做出一个像样的标准后端项目

建议功能：

- 用户登录
- 角色与权限
- 用户管理
- 菜单或资源管理
- 操作日志
- 文件上传
- 分页查询

你在这个阶段重点学习：

- 项目分层
- 接口规范
- 数据表设计
- 权限模型
- 错误响应结构
- 可维护性

### 阶段 5：Node-RED 与物联网平台主线项目（第 11-13 周）

目标：

- 把 `Node-RED` 用到真实后端场景里，而不是只做演示流程

建议功能：

- 设备数据接入
- MQTT 订阅
- 设备状态跟踪
- 阈值告警
- 数据清洗与转换
- HTTP API 对接
- 落库与查询
- 定时任务

你在这个阶段重点学习：

- `Node-RED` 节点组合
- 数据流设计
- 消息结构设计
- 规则处理
- `Node-RED` 与代码型服务协作

### 阶段 6：整合与进阶（第 14 周）

目标：

- 把两条主线连起来，形成完整认知

整合方向：

- `Express` 负责核心业务 API
- `Node-RED` 负责流程编排、设备接入、自动化规则
- 数据库同时服务于后台系统与设备平台
- 使用 `Docker` 编排所有组件

这个阶段你应该能开始回答“系统为什么要这样拆分”，而不是只会跟着教程搭环境。

---

## 5. 重点知识点学习顺序

为了避免初学阶段信息过载，建议按下面顺序学习：

1. `Docker` 基础
2. `Node-RED` 基本操作
3. `JavaScript` 基础语法
4. `Node.js` 运行机制和异步编程
5. `Express` 路由和中间件
6. SQL 和数据库建模基础
7. 管理后台 API 设计
8. `Node-RED` HTTP / Function / Context / Subflow
9. MQTT 与设备消息处理
10. 权限模型、日志、文件上传
11. 事务、索引、性能基础
12. `Node-RED + Express + DB` 协同设计

这个顺序的核心逻辑是：

- 先能跑起来
- 再能写代码
- 再能设计数据
- 最后做系统整合

## 7. 数据库设计强化要求

数据库设计必须单独强化，不能只停留在“会写 SQL”。

你至少要重点掌握这些内容：

- 如何从业务需求中抽取实体
- 如何判断表与表之间的关系
- 如何设计主键和唯一约束
- 如何设计索引而不是乱加索引
- 如何区分“状态表”“日志表”“配置表”“关系表”
- 如何给后台系统设计审计字段
- 如何给物联网场景设计设备表、事件表、告警表、状态表

建议至少完成以下 4 组建模练习：

- 用户、角色、权限模型
- 菜单、资源、接口权限模型
- 设备、设备类型、设备状态模型
- 设备事件、告警记录、规则配置模型

## 8. 每周学习节奏建议

如果你每周能投入 `8-12 小时`，推荐这样安排：

- `2 小时`：基础概念学习
- `3 小时`：样例代码实践
- `3 小时`：主线项目推进
- `1-2 小时`：复盘和笔记整理

每周必须产出至少一个“可运行结果”，例如：

- 一个能启动的独立样例
- 一个新 API
- 一个新数据库模型
- 一条可部署的 `Node-RED` 流程

不要只记笔记，不做可运行产物。

## 9. 本地样例项目组织方式

后续样例代码不写进这份综述文档，而是作为独立项目保存在本地，要求：

- 每个样例都能独立运行
- 每个样例聚焦一个知识点或一组紧密相关的知识点
- 每个样例都带详细注释
- 每个样例都附带启动说明
- 尽量通过 `Docker` 或最少命令运行

建议目录结构如下：

```text
examples/
  01-js-basics/
  02-nodejs-async/
  03-express-routing/
  04-express-middleware/
  05-express-crud-with-db/
  06-db-schema-admin/
  07-db-schema-iot/
  08-nodered-http-flow/
  09-nodered-mqtt-flow/
  10-admin-api-demo/
  11-iot-flow-demo/
```

命名规则建议：

- 编号保证学习顺序清晰
- 每个目录只解决一个核心主题
- 从最小样例逐步走向完整项目

- `Node-RED` 学起来快，但如果没有最小 `Node.js` 基础，后续 Function 节点和接口联调会很吃力

- 用 `Node-RED + Express` 形成完整后端能力
